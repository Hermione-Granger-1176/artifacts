Quick myth-bust
Prompt caching does not save the LLM's response and replay it. Send the same prompt ten times and you will get ten different replies, even when the API reports cached tokens. What actually gets cached are the K and V matrices from the attention mechanism. By the end of this guide, you will know exactly what those are, why they are expensive to compute, and how skipping that computation saves you money and time.

The Big Picture

LLMs are loops, not magic

At the highest level, an LLM is a giant mathematical function that takes a list of numbers in and spits one number out. That output number is a new token. Then it appends that token to the input and runs the whole function again. And again. Until a special "stop" token appears.

prompt = "Explain prompt caching"

tokens = tokenize(prompt)

while (true) {
    embeddings = embed(tokens)

    for each transformer in model:
        embeddings = transformer.attention(embeddings)
        embeddings = transformer.feedforward(embeddings)

    next_token = pick_output(embeddings)

    if next_token == STOP:
        break

    tokens.append(next_token)

print(decode(tokens))

That is the entire inference loop. Every single response you have ever received from ChatGPT, Claude, or any LLM was generated by running this loop over and over, one token at a time. The response streams to you token-by-token because each iteration takes real compute, and showing you partial progress feels much more responsive than making you wait 30 seconds for the full output.

Why the whole prompt goes back in every time
LLMs are stateless. If you only fed in the answer-so-far, the model would forget the question. If you only fed in the question, it would keep trying to produce the first word of the answer. The entire prompt plus all generated tokens so far must go through the model every iteration. This is exactly the redundancy that caching eliminates.

A surprising fact about model code

Despite being called "large" language models, the actual source code of a modern LLM is shockingly short. Open-source models like OLMo 3 can be reimplemented in roughly 200 lines of Python. The "large" part is not the code. It is the billions of numbers (parameters) that the code operates on, all carefully tuned during training.

Step 1 of 4

Tokenizer: text becomes numbers

The tokenizer chops your text into small chunks and assigns each unique chunk an integer ID. These chunks are not always whole words. Common words stay intact, but rare or long words get split into sub-word pieces. This is a design trade-off: too few big tokens means a huge vocabulary table, too many tiny tokens means very long sequences (which cost more to process).

0 tokens 0 words 0 chars

This simulates BPE-style tokenization. Common words stay as single tokens. Uncommon words get split into learned subword pieces. Spaces attach to the following token. Token IDs are deterministic hashes, not real vocabulary lookups.

Why this matters for caching

Tokenization is deterministic. The same text always produces the same token sequence. This is critical for caching: if your second request starts with the same text, the provider knows it will produce the exact same token IDs, which means the same K and V matrices can be reused.

Every model has its own tokenizer
GPT-5 and Claude use completely different tokenizers. Even different versions of GPT can tokenize the same text differently. Tokens are case-sensitive too, because capitalisation carries meaning: "Will" (probably a name) vs "will" (a verb) should be treated differently.

Special tokens you don't see

Beyond your text, LLMs have special tokens for housekeeping. There is a token that signals "I am done talking" (the END token). There are tokens that mark where a user message ends and an assistant message begins, which is how chat models know the conversation structure. You never type these, but they are silently injected into every request.

Step 2 of 4

Embeddings: numbers become meaning

Tokens are just integers. The number 4821 is no more "similar" to 4822 than to 99103. But we need the model to understand that "happy" and "joyful" are close in meaning while "happy" and "submarine" are not. The fix: give every token a position in a high-dimensional space.

An embedding is an array of numbers representing a coordinate in this space. If we had 3 dimensions, an embedding might look like [0.71, -0.33, 0.52]. During training, the model nudges all tokens around this space until words with similar meanings cluster together.

// Created during training, frozen during inference
const EMBEDDING_TABLE = [
    /* token 0 */  [0.12, -0.45, 0.78, ...],  // thousands of dims
    /* token 1 */  [-0.33, 0.91, 0.02, ...],
    ...
    /* token N */  [0.55, -0.11, 0.67, ...],
];

function embed(tokenId) {
    return EMBEDDING_TABLE[tokenId];
}

Why more dimensions help

With just 1 dimension, all tokens sit on a single line. Hard to separate clusters. With 2 or 3, groups start to become visible. Real models use thousands of dimensions (GPT-4 class: 12,000+), allowing the model to capture countless nuances: tone, formality, domain, sentiment, grammar role, all simultaneously.

Position encoding

Order matters. "The cat sat on the mat" means something very different from "the mat sat on the cat." After looking up a token's embedding, the model also encodes the token's position in the sequence into its embedding. This way the model knows not just what each token is, but where it appears.

Step 3 of 4

Attention: the heart of the transformer

This is where the real intelligence lives. The attention mechanism answers one question: "For every token I am about to generate, how much should I care about each previous token?" It does this by computing a set of weights that mix the embeddings together proportionally.

The three projections: Q, K, and V

The model has three weight matrices learned during training: WQ, WK, and WV. Each one transforms the embeddings into a different "view":

Q
"What am I looking for?"
Query
K
"What do I contain?"
Key
V
"What info should I pass along?"
Value

Q and K determine the weights (how much attention to pay). V carries the content that actually gets blended. Remember this split, it is the entire basis of caching.

Why mask future tokens?
When generating the 3rd token, we only know tokens 1 and 2. If we let token 3 see token 4 during training, it would learn to "cheat" by reading ahead. The triangular mask sets future positions to negative infinity. After softmax (which does e^x on each value), e^(-infinity) = 0, so future tokens contribute zero weight. If we used regular zero instead, e^0 = 1, which would still contribute weight. That is why negative infinity specifically, not zero.

What are WQ, WK, WV actually doing?
Honestly, we do not fully know. They are learned during training and end up encoding some transformation that helps the model produce good output. Interpreting what is inside them is an open area of research called "mechanistic interpretability." Each number in these matrices is one "parameter." When a model is described as having 175 billion parameters, these are the numbers being counted.
Step 4 of 4

The KV Cache: this is what gets saved

Here is the core insight. Look at the attention grid above. Play the inference loop in your head: first we process "the", then "the cat", then "the cat sat", and so on. Each time, the weights for all previous tokens stay exactly the same. Row 2 is always the same, whether we are processing 3 tokens or 30. We are redoing huge matrix multiplications that produce identical results.

The fix is two changes to the inference loop:

1. Cache the K and V matrices from each iteration.
2. Only feed the newest token into the model, not the entire prompt.

Without cache vs. with cache

Toggle to see the difference in how much computation each approach requires.

Computation for generating token 5

// Without cache: feed ALL tokens every time
Q = all_embeddings * WQ    // 5 rows computed
K = all_embeddings * WK    // 5 rows computed
V = all_embeddings * WV    // 5 rows computed

// With cache: only compute the new token's row
Q_new = new_embedding * WQ   // 1 row
K_new = new_embedding * WK   // 1 row
V_new = new_embedding * WV   // 1 row

K = append(cached_K, K_new)  // reuse 4 rows, add 1
V = append(cached_V, V_new)  // reuse 4 rows, add 1
That is it. Those K and V matrices are the "1s and 0s" that get cached.
The entire magic of prompt caching comes down to: skip the matrix multiplications for embeddings * WK and embeddings * WV on tokens you have already processed. Fewer multiplications = less GPU time = cheaper and faster.
In Practice

How providers handle it differently

Within a single request, KV caching is standard. Every LLM does it. The clever part that providers like OpenAI and Anthropic do is keeping the cache between requests. If your second request shares the same prefix as the first, the provider reuses the cached K and V instead of recalculating them. But the two providers take very different approaches.

ControlFully automatic. You do nothing, the system tries to route you to cached data.Explicit. You mark which parts of the prompt to cache using cache_control breakpoints.

Temperature does not invalidate the cache
Parameters like temperature, top_p, and top_k all affect the final token selection step, which happens after the attention mechanism has already produced its output. So you can change randomness settings freely between requests without losing your cached K and V.
